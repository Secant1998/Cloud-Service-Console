# Cloud Service Console UI Agent Spec

这份文档是当前项目的 UI 长期规范真身，供后续 agent 直接阅读。

它替代和吸收了早期放在桌面上的 UI 需求草稿。今后如无特殊说明，应以这份文件为准。

## 1. 这份规范的用途

适用范围：

- 当前 `Cloud Service Console` 小程序
- 以后在这套工程里继续做的新页面、新卡片、新工具界面
- 以后如果继续做新的桌面小程序，也可以把这份规范作为默认风格基线

目标：

- 让新 agent 在没有历史对话上下文时，也能直接理解用户的长期 UI 偏好
- 减少“功能做对了，但观感完全不对”的情况
- 减少因为大改结构导致的交接成本

## 2. 总体风格方向

用户偏好的不是传统后台，也不是极简空壳，而是：

- 现代桌面客户端感
- 轻量 SaaS Dashboard 气质
- 玻璃感、渐变、柔和光感
- 结构清楚、信息层级明确
- 可以长期使用，不像 demo

关键词：

- 圆角
- 卡片
- 留白
- 渐变
- 柔和阴影
- 轻微动效
- 信息层次清晰
- 手感顺滑
- 工具感但不粗糙

## 3. 用户明确偏好

### 3.1 视觉

用户长期明确喜欢：

- 大圆角按钮、卡片、输入框、面板
- 浅色模式下偏浅蓝灰、浅白、轻雾感
- 深色模式下偏深蓝灰，而不是纯黑
- 有氛围感的背景，不要平涂死板
- 卡片 hover 时有柔和反馈
- 过渡和动效要丝滑

用户长期明确不喜欢：

- 有棱有角的硬矩形按钮
- 默认后台模板感
- 密密麻麻的工程页
- 强对比硬边框
- 文本被遮挡、换行难看、布局挤压
- 一眼看上去像半成品

### 3.2 交互

用户明确喜欢：

- 开关用滑动式 toggle，不喜欢生硬按钮替代
- 刷新、登录、启停、切换模式都要有明显过程反馈
- 鼠标放到卡片上有轻柔效果
- 背景可有缓慢跟随鼠标的氛围变化

用户明确不喜欢：

- 点了按钮后看起来像卡死
- 完全没有过程反馈
- 过于吵闹或频闪的动画
- 鼠标在哪就一个大光圈那种直白效果

### 3.3 工程约束

用户长期明确要求：

- 尽量少动代码和文件
- 保留旧链路作为兜底
- 改动要便于交接
- 能局部修补就不要大重构

这条要求和 UI 同等重要。

## 4. 浅色与深色主题要求

### 4.1 浅色主题

要求：

- 背景整体要明显偏浅
- 卡片可以是浅色，但不要惨白刺眼
- 输入框内部颜色可以是浅蓝或很浅的冷色，不要纯白到突兀
- 日志区、面板区也可以保持浅色体系
- 当前激活状态必须清晰可辨

### 4.2 深色主题

要求：

- 不要纯黑
- 以深蓝灰、海军蓝、深石墨色为主
- 保持足够层次，不要变成一整块黑面
- 彩色卡片在深色模式下要更克制，不要突兀刺眼
- 当前页面 / 当前激活状态必须明显可辨

### 4.3 主题切换

要求：

- 登录页和主界面都要支持切换
- 主题切换最好即时生效
- 同类内容在切换主题后不应“风格断裂”
- 如果游戏视图有自己的配色，主题切换时也要跟随，而不是必须退出重进

## 5. 登录页规范

登录页不应只是表单摆上去，而应有明显产品感。

必须满足：

- 有独立于主界面的视觉氛围
- 输入框圆角
- 输入内容清晰可见
- placeholder 像 placeholder，不像默认已填值
- 登录进度只在点击登录后出现
- 可以显示连接中的状态文案
- “记住密码”和“自动登录”要做得整洁，非传统笨重 checkbox 风格

当前项目特定要求：

- 默认 `Host` 提示值是 `150.109.100.30`
- 这个值应作为提示，而不是看起来已经填好
- 回车可辅助填入默认 host

## 6. 顶部区域规范

顶部区域是当前产品很重要的一部分，建议长期保持“吸顶 + 轻磨砂 / 轻雾化”风格。

要求：

- 标题区和主操作按钮尽量吸顶
- 滚动时内容不应直接穿过顶部内容，视觉上要自然
- 顶部和正文边缘之间最好有柔和淡入 / 淡出过渡
- 顶部不要重新做成厚重边框大面板

## 7. 左侧导航规范

当前产品结构已经稳定成：

- 左侧固定导航
- 顶部操作条
- 中间内容区

导航要求：

- 当前激活项在浅色和深色主题下都要清晰
- 非激活项不能过于抢戏
- 文字不应轻易换行
- 左下“当前连接”卡片是功能型面板，不只是展示

## 8. 卡片与面板规范

### 8.1 通用卡片

要求：

- 圆角
- 有层次
- 可 hover
- 文字和控件对齐严格
- 尺寸和边距统一

### 8.2 交互反馈

适合的卡片效果：

- hover 轻微抬起
- 柔和变色
- 高光缓慢移动
- 轻微流光

不适合：

- 大幅弹跳
- 颜色剧烈闪烁
- 明显分散注意力的重动画

### 8.3 分隔线

用户对多余分隔线比较敏感。

要求：

- 少而精
- 能去掉第一根就去掉
- 分隔线不要太硬

## 9. 按钮和开关规范

### 9.1 按钮

要求：

- 圆角
- 尺寸统一
- 文字不换行
- hover 有轻微反馈
- Busy 状态清晰

### 9.2 开关

当前项目明确偏好滑动式 toggle。

要求：

- 开和关的视觉位置差异明显
- 如果是异步操作，不要一点击就瞬间跳到完成状态
- 正确做法是：
  - 先进入忙碌态
  - 等云端响应完成
  - 再丝滑滑到目标状态

这个要求当前尤其适用于：

- 云端服务开关
- 视频预览开关
- 控制监视开关
- 主题切换的某些延伸场景

## 10. 刷新、登录、启动、切换模式的反馈规范

凡是耗时操作都必须给出过程感。

包括：

- 登录
- 刷新状态
- 开启监听
- 关闭监听
- 一键配置
- 切换 HTTP / SRT
- 服务启停

推荐反馈：

- 小进度条
- loading 状态
- 忙碌中的文案
- 成功 / 失败后的短提示

不允许：

- 用户点击后界面像冻住
- 无任何阶段提示
- 自动刷新时日志刷屏

## 11. 信息展示规范

### 11.1 连接快照

要求：

- 简洁
- 可点击整体刷新
- 有点击反馈
- IP、地区、服务器信息对齐
- 地区信息尽量友好表达

地区展示偏好：

- 中国大陆尽量到城市级
- 港澳台使用 `China / Hong Kong`、`China / Macao`、`China / Taipei` 这类表达
- 其他国家到国家级即可
- 可配圆形国旗

### 11.2 状态展示

要求：

- 有些状态适合信号灯而不是一长串文字
- 字段要直白，例如 `Active`、`Inactive`、`Autostart`
- 避免冗长技术描述把布局撑坏

### 11.3 活动日志

要求：

- 像受控信息区，不像原始终端输出
- 浅色主题下也可以保持浅底
- 深色主题下保持深底浅字
- 结构清晰，不要太散

## 12. 服务管理视图规范

这是当前小程序最核心的业务页面之一。

要求：

- 云端服务卡片排列整齐
- 启动全部 / 停止全部 / 切换模式这些高频动作要集中、明显
- 视频预览和控制监视属于右侧功能区，但样式要和全局一致
- 不要为了塞信息牺牲对齐和留白

当前项目特定要求：

- `Ingest Bridge` 这个名字已经确定，不要再在 UI 里改回旧称呼
- 切换 `HTTP` / `SRT` 时要显示进度与结果
- 视频预览开关和服务开关都应保持统一 toggle 风格

## 13. 健康检查视图规范

要求：

- 端口检测放在较靠上位置
- 手动检查优先
- 可提供逐个检测和全部检测
- 不做持续监听型高频检查

原因：

- 用户只想在需要时确认端口可达
- 不想给服务器制造长期监测负担

## 14. 玩玩游戏视图规范

这个栏目虽然是“游戏”，但仍然属于主程序的一部分，风格上不能完全脱节。

要求：

- 保持主程序整体质感
- 可以有更强的氛围感
- 但不能因为游戏页让整个程序显得像另一个产品

当前项目特定要求：

- 游戏视图是主程序的一部分，不是外跳网页
- 默认无全局滚动条干扰游戏体验
- 计分板等侧栏可根据空间自适应隐藏或滑出
- 深浅主题切换要影响游戏配色

## 15. 面向当前工程的编码边界

### 15.1 不要把业务后端逻辑放进前端

前端负责：

- 视图
- 交互
- 状态展示
- Canvas / WebRTC / 游戏本地表现层

后端负责：

- SSH
- 远端服务控制
- 云端部署
- 配置判断
- 控制 signaling 桥接
- 坦克游戏云端房间逻辑

### 15.2 保持小改动策略

默认原则：

- 优先改现有文件
- 优先在原有结构里补强
- 不为“看起来更整齐”而大面积拆文件

### 15.3 对旧链路保持敬畏

用户之前多次明确表示：

- 新链路可以接
- 老链路要尽量保留作兜底

所以：

- 不要轻易删除 fallback
- 不要随意删兼容开关

### 15.4 对延迟测试链路保持隔离

坦克游戏里的延迟测试模式有单独边界：

- 首要目标是低延迟 + 丝滑稳定
- 禁止顺手影响视频链路
- 禁止顺手影响正式控制链路

如果要继续优化：

- 优先只改 `TankTroublePanel.tsx`
- 或只改 `tank_trouble_room.py`
- 避免把实验逻辑扩散到主控制台正式功能

## 16. 新 agent 开工前的检查清单

接手这个项目时，先确认：

1. 你读过 `frontend/README.md`
2. 你读过 `frontend/IMPLEMENTATION_LOG.md`
3. 你读过 `frontend/UI_AGENT_SPEC.md`
4. 你知道当前前端真身在 `frontend/src`
5. 你知道后端真身在 `backend`
6. 你知道桌面壳真身在 `src-tauri`
7. 你知道坦克游戏真身在 `TankTroublePanel.tsx`
8. 你知道云端坦克房间真身在 `backend/cloud_setup_bundle/tank_trouble_room.py`
9. 你知道少改文件、保留兜底、不要乱碰其他链路是硬要求

## 17. 修改完成后的最低自检标准

如果你改了 UI，至少自查这些：

1. 浅色和深色都看过
2. 缩放窗口后没有明显挡住、裁切、按钮换行
3. 激活态清晰
4. loading / busy 反馈正常
5. 圆角、间距、对齐没有破坏
6. 不会凭空让 backend 或别的链路失效
7. 如果改了打包相关逻辑，桌面版仍可正常生成

## 18. 这份规范的定位

这份规范不追求“设计理论化”，而追求“以后别再把用户已经反复强调过的偏好做反了”。

如果新需求和这份规范冲突：

- 优先听用户最新明确指令
- 然后再把这份规范更新掉

否则，默认按这份规范执行。
